# aeon-spawn-instance

Multi-agent topology in one call. Given a one-line purpose statement, the skill forks the parent repo, narrows the enabled skill set to what matches that purpose, validates that every enabled skill actually exists on disk, and registers the new instance in `memory/instances.json` so the parent's `aeon-fleet-control` can monitor and dispatch.

Each instance is **inert until the operator manually adds secrets**. There is no secret propagation.

## Inputs

| Param | Format | Description |
|---|---|---|
| `var` | `name: purpose` | e.g. `crypto-tracker: monitor DeFi protocols and token movements on Base`. |

If `var` is empty or unparseable, the skill exits `SPAWN_INVALID_VAR` with no changes.

## What it does

1. **Pre-flight** — `gh auth`, parent identity, rate-limit headroom (≥50 remaining), parse `var` into `NAME` and `PURPOSE`. Derive `REPO_NAME = aeon-${NAME}`.
2. **Fleet check** — read `memory/instances.json`. Refuse if an active duplicate exists; replace if it's archived.
3. **Skill plan** — enumerate the live `skills/` directory (no hardcoded list — that's a known footgun). Keyword-match the purpose against each skill's frontmatter tags and description. Cap at 8 content skills + always include `heartbeat`. Drop any candidate whose source file doesn't exist.
4. **Fork** — `gh repo fork ${PARENT_UPSTREAM} --fork-name ${REPO_NAME} --clone=false`, with fallback to `POST /repos/{owner}/{repo}/forks`. Wait for fork availability (asynchronous on GitHub's side, up to ~30s).
5. **Configure the fork** — clone, write a customized `aeon.yml` with only the planned skills `enabled: true`, generate `SETUP.md` listing every env var the enabled skills reference (parsed from their SKILL.md files), reset `memory/` to a clean state, commit, push.
6. **Enable Actions** — `PUT /repos/{owner}/{repo}/actions/permissions` with a typed boolean (`-F enabled=true`, not `-f` — known footgun).
7. **Post-flight** — fork reachable, HEAD commit present, Actions enabled.
8. **Register** — append to the parent's `memory/instances.json`.

## Purpose → skill mapping

The skill matches purpose keywords against `skills/*/SKILL.md` frontmatter:

| Keywords | Tag / name overlap |
|---|---|
| `crypto`, `defi`, `token`, `chain`, `wallet` | tag `crypto`, name contains `token`/`defi`/`chain` |
| `research`, `paper`, `academic` | tag `research`, name contains `paper`/`research` |
| `social`, `twitter`, `tweet`, `farcaster` | tag `social`, name contains `tweet`/`farcaster`/`reply` |
| `news`, `digest`, `feed`, `brief` | name contains `digest`/`brief`/`morning`/`rss` |
| `dev`, `code`, `github`, `repo` | tag `dev`, name contains `github`/`code`/`repo`/`pr-` |
| `security`, `vuln`, `audit` | tag `security`, name contains `security`/`vuln`/`audit` |
| `prediction`, `polymarket`, `kalshi`, `market` | name contains `polymarket`/`kalshi`/`market`/`narrative` |

If no theme matches, falls back to `[morning-brief, reflect, heartbeat]` (filtered by existence).

## Security model

- **No secret propagation.** Never reads or writes API keys into the child repo. Each instance has its own keys for billing isolation and blast-radius containment.
- **No autonomous activity** from a half-spawned instance — the fork is inert until secrets are added.
- **Recovery, not deletion.** If any phase fails partway through, the fork is left in place with explicit recovery instructions. Operator is the only party authorized to delete.

## Exit taxonomy

| Code | Meaning |
|---|---|
| `SPAWN_OK` | Fork created, configured, registered. |
| `SPAWN_FORK_EXISTS_RECOVERED` | Pre-existing un-registered fork → configured + registered now. |
| `SPAWN_FORK_EXISTS_REGISTERED` | Active duplicate refused. |
| `SPAWN_INVALID_VAR` | `var` empty or unparseable. |
| `SPAWN_NO_SKILLS` | No skills matched the purpose. |
| `SPAWN_FORK_FAILED` / `SPAWN_PUSH_FAILED` / `SPAWN_ACTIONS_FAILED` | Recovery instructions emitted. |

## Generated SETUP.md

The child's SETUP.md is dynamically generated by parsing each enabled skill's SKILL.md for uppercase env-var references (pattern: `[A-Z][A-Z0-9_]{2,}_(KEY|TOKEN|URL|ID|SECRET)`). Operator-required secrets land in the SETUP under three sections: required (Claude auth), recommended (notification channels), and per-skill (env vars the enabled skills actually use).
